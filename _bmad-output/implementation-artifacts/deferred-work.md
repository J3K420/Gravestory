# Deferred Work

## Deferred from: code review of 1-1-run-scan-credits-database-migration (2026-06-05)

- **scan_credits read error silently drops paid credits** — paying user gets `purchased=0` on transient network error; intentional fail-soft design but asymmetric with fail-closed behaviour for scan_events. Revisit when credits are live and user-reported failures surface.
- **TOCTOU race on check-then-insert** — two concurrent browser tabs or rapid taps can both pass the scan limit check before either increments the counter. Fix requires an atomic DB function (INSERT INTO scan_events RETURNING count > limit). Track for post-launch hardening.
- **scan_credits.updated_at has no BEFORE UPDATE trigger** — column stays at INSERT time forever. Add `CREATE OR REPLACE FUNCTION update_scan_credits_updated_at()...` trigger in Story 1.3's migration when the webhook behavior is defined.
- **incrementWebScanCount double-increment on retry** — network timeout after server write causes a second INSERT, over-counting one scan. Fix: pass a client-generated UUID as the scan_events primary key so retries are idempotent.
- **PaywallScreen.js shows base limit (10) not totalAllowance** — progress bar and count text overflow for users with purchased credits. Fix: use `route.params.limit` (already passed by CameraScreen as `scanCheck.limit`) instead of re-deriving from the SCAN_LIMIT_USER constant. [mobile/src/screens/PaywallScreen.js]
- **purchased column has no CHECK constraint** — a negative value from a webhook refund bug would silently reduce allowance below the free tier. Add `CHECK (purchased >= 0)` in Story 1.3's migration DDL or a separate follow-up migration.

## Deferred from: code review (2026-06-29) — expo-media-library removal for Play Store photo-permissions compliance

- **Silent auto-pin → manual-pin downgrade for gallery-picked photos has no user-facing signal** — Removing expo-media-library means a library-picked gravestone photo on Android no longer auto-pins from recovered EXIF GPS; it silently falls through to manual pin placement. Behavior is correct and the manual drag-to-correct flow already exists, but a user who previously got an auto-pin may experience the absence as "the pin is missing/wrong." Consider a one-time gentle hint ("Tap the map to place this grave") when a library pick yields no location. Product decision, not a code defect. [mobile/src/screens/CameraScreen.js]
- **8-line rationale comment sits inside the app.config.js plugins array** — The expo-media-library plugin entry was replaced with a multi-line prose comment explaining the removal, wedged between two real plugin entries. Low risk, but a future array reformat/edit could detach or drop it, losing the institutional rationale. Optionally relocate the long-form rationale to the commit message / a CHANGELOG and keep a one-liner in-array. Maintainability nit. [mobile/app.config.js]

## Deferred from: code review (2026-07-08) — Chinese/foreign-language gravestone support (items 1-4)

- **`ancestral_origin` has no dedicated DB column** — it round-trips only via the biography TEXT (the bio prompt now MUST name it, and the stone-only fallback embeds it), not as a structured field; it is dropped from the structured story object on cloud save→reload. Verified honest (comment at CameraScreen.js story-build) and no downstream reader expects the structured field to survive (GEDCOM export uses `story.location`; the public redactor operates on bio text). Acceptable for a prompt-only pass. If a future feature needs the structured homeland (e.g. a dedicated GEDCOM birthplace field or a filter), add a `graves`/`stories` column + map it in `sync.js` storyToRow/rowToStory. [mobile/src/screens/CameraScreen.js, mobile/src/lib/sync.js]
- **Pre-existing: `'urn'` symbol key substring-matches "b**urn**er"** — surfaced (not caused) by the new `'incense'` example: an OCR symbol string "incense burner" matches the pre-existing `'urn'` key and injects the funerary-urn gloss. Low harm (thematically adjacent), and it predates this change. Fix if the symbol table is ever reworked to word-boundary matching rather than bare `.includes()`. [mobile/src/lib/biography.js buildSymbolContext]
- **Pre-existing: ASCII-only public name-filter is blind to CJK personal names** — `filterMentionsForPublic` matches `/\b([A-Z][a-z]+…)/`, so a Han-script personal name left in a mention sentence would not be caught by the living-name redaction. Not caused by this diff and net-safer now (bios are forced to English); the honorific glyphs the LANGUAGE directive permits in parentheses (e.g. 顯考) are titles, not names. Revisit if/when public bios routinely carry CJK personal names. [mobile/src/lib/api-gemini.js filterMentionsForPublic]
