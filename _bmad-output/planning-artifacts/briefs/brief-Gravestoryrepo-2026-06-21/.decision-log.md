# Decision Log — Option B Background Scanning brief

Canonical memory + audit trail. Every decision, change, and override recorded as the conversation unfolds.

## 2026-06-21 — Session start
- Intent: **Create** a product brief for "Option B — Background Scanning".
- Pre-decided by user (carried in from the originating conversation):
  - **Approach DECIDED**: move research+biography pipeline server-side onto the existing Cloudflare Worker. Phone uploads photo+GPS; Worker runs pipeline; result written to Supabase.
  - **Notification model DECIDED**: FULL PUSH (works when app is fully closed) via expo-notifications + push tokens. Requires a new native build (not OTA).
  - **BMAD flow chosen**: Product Brief → PRD → Architecture → Epics & Stories → **Validation (implementation-readiness)** → Dev. User explicitly required a validation phase after story creation.

## 2026-06-21 — Discovery Q1-Q3 (initial answers, being pressure-tested)
- **Q1 Cost model**: charge on SUCCESS, both client- and server-side. (A failed scan does NOT consume a scan/credit.) — NOTE: this REVERSES today's behavior where a scan is counted right after OCR succeeds regardless of bio outcome. Flagged for pressure-test: defines what "success" means + double-charge race.
- **Q2 Coexistence**: foreground (synchronous) and background pipelines COEXIST long-term — not a full migration. Building a parallel path.
- **Q3 Launch dependency**: Option B is a LAUNCH BLOCKER. Rationale: it mitigates the damage of the long loading screen — a user who leaves and returns to a PAUSED scan (instead of a finished one) reads as broken ("wtf is this" vibe). Background completion turns the long wait from a liability into an acceptable experience.

## 2026-06-21 — Discovery Q1-Q3 RESOLVED (final, supersedes above)
- **Q1 Cost model — FINAL**: Charge is bound to **OCR success**, exactly as today (NOT bio-success). So the scan is counted once OCR confirms a readable gravestone, regardless of whether downstream research/bio succeeds. This REVERSES the earlier "charge on success" lean and keeps current behavior. Key consequence: this naturally caps cost-abuse (you can't burn unlimited Tavily/Gemini without consuming scans), so NO separate job-spam throttle is needed. The charge happens server-side at the OCR step (moved from client, a security improvement). A failed *bio* after a successful OCR still counts — same as today.
- **Q2 Routing — FINAL**: **Automatic.** EVERY scan becomes a server-side background job. The foreground "loading" screen is just a live *watcher* of that job. A fast scan still feels instant (watcher sees it finish in seconds); a slow scan survives the user backgrounding/closing the app (job keeps running server-side) and pings via push when done. No user toggle. This is the single biggest architecture simplifier — one pipeline path (server), the phone is always just an observer.
- **Q3 Launch — FINAL**: LAUNCH BLOCKER confirmed. Timeline: ~1 week until launch (as of 2026-06-21), so ~targeting launch ≤ 2026-06-28. Cheap-mitigation fallback (a "still working…" foreground state) was considered and DECLINED in favor of doing Option B properly within the week.
- **Q4 Expo push cost — RESOLVED**: Expo push notifications are FREE (no per-message/volume fee; relays to free FCM/APNs). Zero ongoing cost. Only cost is one-time eng + a new native build (expo-notifications is native). Green-lit to proceed.
- **Q5 Scope (post-draft)**: Web = OUT (ignore). iOS = OUT for now ($100/yr Apple dev acct cost not paid). Android-only.

## 2026-06-21 — DECISION REVERSAL: Option B SHELVED for launch
- **Trigger**: User is on the Cloudflare FREE tier. Option B's durable 40s background job needs the clean queue/durable-object pattern, which requires Cloudflare Workers PAID ($5/mo for Queues; Durable Objects historically paid too). A 40s synchronous job held open in a single free-tier Worker request is fragile (wall-clock duration + request-model mismatch; CPU-time 10ms is survivable since the pipeline is network-wait-bound, but the duration/request model is the real problem).
- **DECISION**: Put full Option B (server-side pipeline + push) ON THE BACK BURNER. Do NOT build it for launch.
- **REPLACEMENT (what we build instead, now)**: Add a small warning/infographic to the LOADING screen telling users to stay on the page while it works — and reassuring them that if they DO switch apps, the scan will resume where it left off when they return. This reframes the existing freeze-on-background behavior as expected rather than broken (kills the "wtf is this" vibe) at ~1% of the cost. Pure OTA, no build, no CF cost.
- **STATUS of this brief**: Option B brief is preserved as a SHELVED design (revisit if/when on Workers Paid). The BMAD flow (PRD/architecture/stories) for Option B is paused indefinitely. The active task is now the small loading-screen reassurance UI.
