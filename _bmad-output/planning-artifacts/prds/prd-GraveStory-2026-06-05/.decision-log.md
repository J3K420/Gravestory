# Decision Log — GraveStory PRD

> Canonical memory and audit trail. Every significant product decision recorded here.
> Retroactively populated from CLAUDE.md and project-context.md on 2026-06-05.

---

## DL-1: No framework / no build step on web (Phase 1)
**Decision:** Web UI is vanilla HTML/CSS/JS loaded as classic `<script>` tags. No npm, no bundler, no TypeScript, no ES modules.
**Rationale:** Maximizes deployment simplicity (static files on GitHub Pages, no CI/CD required). Eliminates dependency management overhead for a solo developer. The trade-off is no tree-shaking, no type safety, and manual load-order management — accepted.
**Status:** Final.

---

## DL-2: Cloudflare Worker as the only proxy (Phase 1)
**Decision:** All calls to Gemini, Tavily, WikiTree, and R2 route through a single Cloudflare Worker. No API keys in client code.
**Rationale:** Prevents key exposure in static client files. Centralizes auth and rate-limit logic. Worker is cheap for this traffic volume.
**Status:** Final. Worker enforces two auth layers: ALLOWED_ORIGIN (browser) + CLIENT_KEY header (mobile/direct).

---

## DL-3: Supabase for auth + DB (Phase 1)
**Decision:** Supabase (PostgreSQL + Auth) as the cloud backend.
**Rationale:** Managed Postgres with row-level security, built-in auth (Google OAuth + email), generous free tier. RLS policies enforce data isolation without server-side logic.
**Status:** Final.

---

## DL-4: Soft-delete only for stories (Phase 1)
**Decision:** Stories are never hard-deleted. `deleted_at` timestamp propagates deletions via delta sync.
**Rationale:** Enables reliable multi-device sync without complex tombstone management. Deletion propagates naturally via the `updated_at` watermark sync.
**Status:** Final.

---

## DL-5: Gemini as the sole AI model; fallback to gemini-2.5-flash (Phase 2)
**Decision:** Primary model is `gemini-3.1-flash-lite`; auto-fallback to `gemini-2.5-flash` on 503/429/overload.
**Rationale:** Flash-lite is cheaper and fast enough for OCR and standard biographies. 2.5-flash as fallback avoids hard failures during load spikes. Temperature 0.1 for deterministic output.
**Status:** Final.

---

## DL-6: Stone-only fallback when research returns nothing (Phase 2)
**Decision:** If Tavily and WikiTree both return empty, generate a short narrative from the inscription alone — without calling Gemini.
**Rationale:** Prevents AI hallucination about unknown individuals. The model cannot be trusted to invent plausible-sounding facts about a person with no retrievable record.
**Status:** Final. Critical invariant — do not remove.

---

## DL-7: Parallel research execution (Phase 2)
**Decision:** Tavily, WikiTree, Wikidata, Chronicling America, Wikipedia (article + portrait) all fire in parallel after OCR.
**Rationale:** Total pipeline latency equals the slowest source, not the sum. Acceptable trade-off: slightly higher peak concurrency load on the Worker.
**Status:** Final.

---

## DL-8: Expo managed workflow for mobile (Phase 2)
**Decision:** React Native via Expo managed workflow. No bare ejection.
**Rationale:** Keeps the mobile codebase maintainable for a solo developer. EAS Build handles native compilation without local toolchain setup. Trade-off: some native module constraints (e.g., Overpass queries blocked by Cloudflare Worker IPs on mobile).
**Status:** Final. Bare ejection explicitly out of scope.

---

## DL-9: Separate web and mobile codebases (Phase 2)
**Decision:** Web (`js/`, `index.html`) and mobile (`mobile/src/`) are parallel codebases with no shared code (aside from conceptual parity).
**Rationale:** Web uses classic scripts (no ES modules); mobile uses ES modules. The runtime environments are incompatible at the module level. Shared logic is duplicated by convention, not by import.
**Status:** Final. Any pipeline logic change must be applied to both.

---

## DL-10: Credits-only monetization; no subscriptions (Phase 9)
**Decision:** Freemium with one-time credit packs (5/$0.99, 20/$2.99, 60/$6.99). No unlimited subscriptions.
**Rationale:** Unlimited subscriptions create unbounded Tavily API cost per user. Credits align revenue directly to cost. The risk of the subscription model was surfaced in Phase 9 planning and explicitly rejected.
**Status:** Final.

---

## DL-11: Lifetime (not monthly-reset) free scan limit (Phase 9)
**Decision:** Free scan limits are lifetime one-time (not monthly reset).
**Rationale:** Monthly resets allow every user to consume 10 Tavily scans/month for free indefinitely — unacceptable cost at scale. Lifetime trial model: new users get a generous enough sample to evaluate the product, then purchase if they want more.
**Status:** Final. Changed from monthly-reset design mid-Phase 9 (Session 2).

---

## DL-12: Scan limits fail closed on Supabase error (Phase 9, Session 3)
**Decision:** If the Supabase scan-count read fails, the scan is blocked (not allowed).
**Rationale:** Fail-open is a security/cost vulnerability — any Supabase outage would allow unlimited free scans. Fail-closed is the conservative default; users see a connection error rather than accidentally getting free scans.
**Status:** Final.

---

## DL-13: No Overpass on mobile for grave node search (Phase 8b)
**Decision:** Mobile uses two-pass Nominatim + Photon for grave node search instead of Overpass.
**Rationale:** All Overpass mirrors return 403/406 to Cloudflare Worker IPs and directly from the React Native HTTP stack. Replaced with Nominatim viewbox-biased search (without `bounded=1`, which applies an importance-score threshold that silently drops low-importance grave nodes) and Photon (Elasticsearch-backed, indexes low-importance named nodes better).
**Status:** Final. Do not attempt to re-add Overpass on mobile.

---

## DL-14: No cemetery boundary polygon on mobile (Phase 8b)
**Decision:** Mobile does not draw a cemetery boundary polygon.
**Rationale:** Nominatim `polygon_geojson=1` approach was implemented and tested; produced incorrect boundaries (e.g., Machpelah Cemetery polygon did not contain Houdini's actual grave node). Removed entirely. Web retains Overpass-based boundary drawing.
**Status:** Final. Do not re-add boundary drawing on mobile.

---

## DL-15: Floating overlay instead of `<Callout>` for map markers on mobile (Phase 3 / Phase 8)
**Decision:** Map marker interaction uses state-driven floating `View` overlay, not react-native-maps `<Callout>`.
**Rationale:** `<Callout onPress>` with custom child Views silently swallows touch events on Android. The overlay pattern (tapping marker sets `selectedStory` state; tapping map or ✕ dismisses) is reliable across both platforms.
**Status:** Final. Do not replace with `<Callout>`.

---

## DL-16: Historical figure exception requires Wikipedia AND date alignment (Phase 8g / Phase 9)
**Decision:** The extended historical-figure biography (up to 2,500 words) fires only when ALL three conditions hold: dates within ±5 years, a Wikipedia article in numbered sources, and every claim carrying an `[N]` marker.
**Rationale:** The namesake guard prevents "John Adams d.1931" from inheriting the Founding Father's biography. Without the Wikipedia source condition, the model relied on recalled training knowledge rather than retrieved evidence — violating the "memory is not a source" invariant.
**Status:** Final.

---

## DL-17: `tracksViewChanges` pattern for custom SVG map markers (Phase 8)
**Decision:** `GraveMarker` component starts with `tracksViewChanges={true}`, then flips to `false` via `onLayout`.
**Rationale:** react-native-maps takes a native bitmap snapshot of the marker. Setting `tracksViewChanges={false}` unconditionally causes the snapshot to be taken before SVG finishes painting, making the marker disappear. Starting `true` and flipping on layout ensures the snapshot captures the rendered SVG.
**Status:** Final.
