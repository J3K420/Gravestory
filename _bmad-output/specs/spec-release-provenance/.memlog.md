topic: Immutable build release and run provenance
updated: 2026-07-14T21:06
---

- (capability) CAP-1: Each web, Worker, mobile build, and mobile OTA release can be tied to a reviewed commit and unique platform release identifier.
- (capability) CAP-2: Preflight tooling refuses an unsafe source state and produces a reviewable release manifest without deploying.
- (capability) CAP-3: Rollback procedures identify the preceding immutable release and component-specific constraints.
- (constraint) Respect the Cloudflare Pages allowlist, Expo channel and runtime rules, migration ordering, and current legacy-origin retirement gates.
- (constraint) No production release is executed or mutated by repository verification tooling.
- (non-goal) No replacement of EAS, Cloudflare, Supabase, RevenueCat, or Play Console release systems.
- (event) Coherence validation passed.
- (event) Preservation validation passed against the program audit and current cutover runbook.
- (event) Spec finalized after coherence and preservation validation.
- (decision) Release provenance is append-only and two-phase: immutable candidate preflight first, then a new immutable final record referencing it after an approved external release supplies platform evidence.
- (decision) Database-only releases use migration identifiers, checksums, and approved execution evidence; they do not require a fake platform release ID.
- (capability) CAP-4: Finalization rejects stale candidate baselines when another component release has completed since preflight.
- (constraint) Review evidence is explicit input: reviewed commit, PR identity or equivalent versioned attestation, and completed BMad review record; absence makes preflight ineligible.
- (constraint) Depends on the deploy-configuration and Supabase change-control outputs.
- (decision) Immediately before an approved external release, operators rerun a non-deploying candidate-baseline validation; stale candidates cannot drive the release.
- (decision) After external execution begins, finalization always appends execution evidence; concurrent-baseline conflict is recorded and the release is not promoted as current.
- (decision) Migration identity hashes Git blob bytes, and unverified live state is labeled repository-intended rather than asserted compatible.
- (capability) CAP-5: Release record IDs and content hashes plus history validation detect candidate/final mutation or deletion.
- (constraint) Review evidence is cryptographically or structurally bound to the exact candidate source SHA and PR or versioned review artifact.
- (constraint) Web release preflight rejects cached-asset changes without the required service-worker cache version change.
- (decision) Supersedes the earlier statement that finalization rejects every stale candidate: staleness blocks before execution; after execution starts, finalization always appends evidence and marks any conflict.
- (decision) An append-only execution-intent lease on current origin/main is created before the external command; only its candidate may execute for that component, and finalization is idempotent by execution ID.
- (decision) The first record for a component uses an explicit genesis baseline from repository evidence or an approval-bound platform observation; unknown rollback state blocks ordinary eligibility unless the owner accepts a no-known-rollback genesis.
- (decision) Orphaned execution intents never expire into overlap: abandon requires evidence no execution began; otherwise reconcile from execution evidence or remain blocked pending explicit owner decision.
- (event) Final BMad Blind Hunter, Edge Case Hunter, and Acceptance Auditor re-review passed after triage.
