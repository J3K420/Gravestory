topic: Runtime processes logging parity and operational applicability
updated: 2026-07-14T21:06
---

- (capability) CAP-1: Operational documentation maps process, port, concurrency, and disposability responsibilities to Cloudflare Pages, Cloudflare Workers, Expo clients, and local admin tools.
- (capability) CAP-2: Server-side logs are emitted as bounded event records without relying on local files or exposing sensitive payloads.
- (capability) CAP-3: Supported development and verification paths state which production backing-service semantics they reproduce and which parity gaps remain.
- (capability) CAP-4: Intentional client persistence and offline queues have documented crash, retry, idempotency, and ownership boundaries.
- (constraint) Do not add a server process, queue, scheduler, or log-drain vendor unless a demonstrated GraveStory requirement needs one.
- (constraint) Client-device storage is product state and is not treated as a violation of the server-process statelessness factor.
- (non-goal) No production observability integration or retention-policy change without approval.
- (event) Coherence validation passed.
- (event) Preservation validation passed against the program audit and repository logging discipline.
- (event) Spec finalized after coherence and preservation validation.
- (capability) CAP-5: The final batch reconciles the baseline audit and classifies every original gap as implemented, compliant, platform-managed, approval-blocked, or explicitly deferred.
- (decision) Background-runtime coverage includes the web service-worker cache lifecycle and mobile local completion notifications as well as pending scans and sync.
- (decision) Parity covers every configured EAS profile and channel, including development, preview, phase9, and production.
- (constraint) The bounded redaction contract covers existing and future Worker log events.
- (capability) CAP-6: Worker upstream calls have bounded timeouts and mutating routes have duplicate-delivery idempotency tests where retries can occur.
- (decision) Parity follows a ladder of repository checks, local or emulated same-type services, existing preview profiles, and only then optional remote staging when provider-specific integration risk justifies its cost.
- (constraint) Depends on all preceding implementation batches.
- (decision) Every mutating Worker route is inventoried and classified for duplicate-delivery behavior; idempotency tests or explicit exception evidence are required, including upload-image.
- (event) Final BMad Blind Hunter, Edge Case Hunter, and Acceptance Auditor re-review passed after triage.
