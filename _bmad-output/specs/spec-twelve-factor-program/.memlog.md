---
topic: GraveStory Twelve-Factor architectural hardening program
updated: 2026-07-14T21:06
---

- (direction) Use the official Twelve-Factor repository as the architectural standard and audit GraveStory before changing application code.
- (direction) Inspect GraveStory own stack, deployment platforms, environments, database migrations, secrets, logs, background work, and release workflow; do not copy another project blindly.
- (constraint) Do not deploy production, mutate live data, rotate secrets, retire legacy services, or perform destructive operations without explicit approval.
- (constraint) Preserve unrelated work by using isolated branches and keep each batch independently reviewable and shippable.
- (decision) Apply the factors per deployable component inside the existing product monorepo; document the strict one-codebase exception instead of splitting the repository.
- (capability) CAP-1: Produce an evidence-backed assessment of all twelve factors with applicable, partial, compliant, and platform-managed outcomes.
- (capability) CAP-2: Rank gaps by risk and dependency order so foundational verification and safety work precede broader refactoring.
- (capability) CAP-3: Define small branch contracts with explicit acceptance criteria, verification, dependencies, and production boundaries.
- (capability) CAP-4: Implement every applicable factor through the branch contracts, with tests and operational documentation.
- (capability) CAP-5: Run the actual BMad code-review workflow on every code-change batch and fix or explicitly triage every finding.
- (capability) CAP-6: Verify, commit, push, open a PR, merge, and resync main before beginning the next batch.
- (assumption) The official next branch is the requested standard because the upstream repository identifies it as the updated manifesto under active governance.
- (event) Coherence validation passed: capabilities are testable, constraints bend implementation, and production mutations are excluded.
- (event) Preservation validation passed: every load-bearing user requirement is represented in the kernel or batch roadmap companion.
- (event) Spec finalized after coherence and preservation validation.
- (decision) Full-spec BMad code reviews include Blind Hunter, Edge Case Hunter, and Acceptance Auditor; no-spec reviews omit only the auditor.
- (decision) The final runtime-operations batch reconciles the baseline audit and records the terminal status of every gap.
- (decision) Deploy-configuration boundaries precede release-provenance implementation because release records consume configuration identity.
- (decision) No factor requires a complete duplicate remote staging graph; GraveStory uses the smallest parity mechanism that proves each risk, and remote staging remains optional and approval-bound.
- (decision) A shared evidence companion records the pinned upstream standards and file-level GraveStory evidence for all twelve factors; sources arrays remain reserved for fully absorbed file inputs.
- (decision) Program completion permits evidence-backed approval-blocked or explicitly deferred outcomes when repository work cannot cross an owner gate.
- (event) Final BMad Blind Hunter, Edge Case Hunter, and Acceptance Auditor re-review passed after triage.
