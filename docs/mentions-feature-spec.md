# GraveStory — "Mentions" Feature Spec (implementation-ready)

A bottom-sheet on the Result screen ("Mentions" / "Also found in…") that surfaces short, Gemini-written, name-safe one-sentence pointers to the raw research hits the pipeline currently fetches and discards (Tavily, Chronicling America, Internet Archive, Wikipedia). Each sentence is a single hyperlink to its source. It mirrors the existing tappable symbol-chip → bottom-sheet pattern, is generated by ONE batched Gemini call (non-fatal, null-guarded, NOT scan-limit-gated), persists on the story like `symbol_meanings`, and shows on the user's own story AND on public/global-map stories. **The bio is unchanged; mentions do NOT feed the bio.**

This is the exact precedent set:
- `symbol_meanings` (migration 010, `resolveSymbolMeanings`) — the batched-call + jsonb-column + persistence pattern.
- The kinship kernel (migration 021, `subjects`/`relationships`) — attaching structured non-bio data lifted from transient pipeline vars onto `currentStory` so it round-trips.
- `redactLivingNamesForPublic` (S62, `js/api-gemini.js`) — the no-living-relative-names instruction we reuse.

## 1. Data model

### New field
- **Field name:** `mentions`
- **In-memory shape (top-level story property, NOT nested under `graveData`):** `Array<{ sentence: string, url: string, source: string, year: string|null }>`
  - `sentence` — the Gemini-authored, name-safe one-liner (e.g. `"Mentioned in a 1919 obituary in the Aiken Courier."`). This is the link label.
  - `url` — the source URL the sentence links to (must pass `startsWith('http')` to be clickable).
  - `source` — a short machine hint of which pipeline source produced it (`'web' | 'chronicling' | 'archive' | 'wikipedia'`). Used only for ordering/debug; never injected raw into UI without escaping.
  - `year` — best-effort 4-digit string parsed from the title/date, or `null`. Stored so a dead link still reads sensibly (link-rot mitigation, §7).
- **Empty/falsy → stored as `null`** (matches `symbol_meanings` exactly). Because `mentions` is an array, use the array-aware null treatment used for `subjects`/`relationships` so empty arrays don't write `[]` into jsonb (see storyToRow below).

## 2. Generator

### Function
- **Name:** `resolveMentions(rawHits, subjects)`
- **Web file:** `js/api-gemini.js` (sibling of `resolveSymbolMeanings` at `js/api-gemini.js:212`).
- **Mobile file:** `mobile/src/lib/api-gemini.js` (ES-module port; adds the 30s `fetchWithTimeout` per Gemini call, same as `resolveSymbolMeanings` there).
- **Inputs:**
  - `rawHits` — normalized array of `{ url, kind, snippet, year }` aggregated from the per-source vars before they go out of scope (§3). `kind` is a human phrase the model can paraphrase (e.g. `"a present-day web page"`, `"a Chronicling America newspaper page"`, `"an Internet Archive book/county history"`, `"a Wikipedia article"`).
  - `subjects` — `graveData.subjects` (the deceased allowlist), so the model knows which names it is permitted to keep (reused from the S62 redaction allowlist mechanic).
- **Model:** inherits `geminiCallWithFallback` (`js/api-gemini.js:18`): PRIMARY `gemini-3.1-flash-lite`, FALLBACK `gemini-2.5-flash`, `generationConfig: { temperature: 0.1, responseMimeType: 'application/json' }` (no `responseSchema` — validate in code, exactly like `resolveSymbolMeanings`).

### Batched prompt (one call)
The prompt mirrors `resolveSymbolMeanings`' precheck/batch/null-guard discipline AND embeds the S62 no-living-relative rule verbatim. Build an allowlist block from `subjects` as `- Name (birth_date–death_date)`.

```
You are writing short pointer sentences for a gravestone-history app. Each item below is a
real research hit ABOUT one or more DECEASED people. For each item, write ONE short, natural
sentence a museum visitor would read, telling them WHERE this person is mentioned — e.g.
"Mentioned in a 1919 obituary in the Aiken Courier." or "Appears in a county history from 1908."
Use the item's source kind and year. Do NOT summarize the content, quote it, or add facts not
present in the snippet. Keep it under ~18 words. No citation markers, no quotation marks.

THE DECEASED SUBJECT(S) — you MAY name these people exactly as written:
${allowed.length ? allowed.map(n => `- ${n}`).join('\n') : '- (none listed — infer the subject and keep only that person\'s name)'}

NAME-SAFETY RULES (follow EXACTLY):
- For ANY person mentioned in a snippet who is NOT one of the deceased subjects above and is
  NOT confirmed dead by an explicit death year in the snippet, do NOT name them — refer to them
  by relationship only (her son, his wife, a daughter), or omit them.
- If unsure whether a named person is living or dead, treat them as LIVING and do not name them.
  When in doubt, remove the name.
- A person explicitly shown as deceased in the snippet (a death year, "the late", "predeceased")
  is NOT living — you may name them.

QUALITY RULES:
- If a snippet does NOT clearly refer to one of the deceased subjects above (e.g. it is about a
  different person who happens to share the surname), return null for that item. An honest null is
  better than pointing the visitor at the wrong person.
- If you cannot write a confident, useful sentence, return null for that item.

Return ONLY a JSON object whose keys are the item ids given below and whose values are the
sentence string, or null. Example shape:
{ "m0": "Mentioned in a 1919 obituary in the Aiken Courier.", "m1": null }

Items:
${rawHits.map((h, i) => `- m${i}: kind=${h.kind}; year=${h.year || 'unknown'}; snippet="${(h.snippet||'').slice(0,400)}"`).join('\n')}

Return only JSON.
```

(The two load-bearing reused S62 lines: *"replace any person NOT confirmed dead (no explicit death year) and NOT in the allowed-deceased list with their relationship only"* and *"If unsure whether a named relative is living or dead, treat them as LIVING and generalize the name. When in doubt, remove the name."* — quoted from `redactLivingNamesForPublic` in `js/api-gemini.js:374-428`.)

### Output / validation (mirror `resolveSymbolMeanings:245-266`)
1. Early return `[]` if `rawHits` is not a non-empty array.
2. Cap input to a small number of hits before the call (see §7 cap — e.g. first 8 after relevance filtering).
3. ONE `geminiCallWithFallback` batch.
4. If `!data || data.error || !data.candidates?.[0]?.content?.parts?.[0]?.text` → return `[]`.
5. `safeParseJSON(text, {})`; if not an object → return `[]`.
6. NULL-GUARD: iterate parsed `{ m{i}: sentence|null }`; for each, keep only when `typeof v === 'string' && v.trim()` AND the matching `rawHits[i].url` passes `startsWith('http')`. Build `{ sentence: v.trim(), url, source: hit.source, year: hit.year || null }`.
7. De-dupe by `url` (keep first).
8. Wrap everything in `try/catch` → `console.warn('resolveMentions failed (non-fatal):', …); return [];`. **Fails CLOSED to `[]`** (omit the sheet), exactly like `resolveSymbolMeanings` fails closed to `{}`.

### Where it runs in runPipeline (`index.html`)
- **Start early, mirroring `symbolMeaningsPromise`:** but unlike symbols, mentions needs the raw research arrays, which only exist inside the `else` research branch. So build the normalized `rawHits` and kick off the call at **`index.html:~1376`**, immediately after the `RESEARCH_YIELD` `logEvent` (`index.html:1369-1377`) and BEFORE the branch closes at line 1378 — this is the last point where `searchResults`, `chronResults`, `archiveResults`, `wikiSummaryResults`, `mergedSearchResults` are all in scope.

  ```js
  mentionsPromise = resolveMentions(mentionHits, graveData.subjects)
    .catch(e => { console.log('resolveMentions failed (non-fatal):', e?.message || e); return []; });
  ```
- **Hoist `let mentionsPromise = Promise.resolve([]);`** near `let _origUnique = [];` (`index.html:1146`) and `let wikidataResult = null;` (`index.html:1109`), so the cache-hit branch (`index.html:1147-1168`) leaves it as the resolved-empty default (no fresh research → no mentions, identical handling to `_origUnique = []`).
- **Await + attach** alongside the symbol-meaning merge at `index.html:1385-1388`:
  ```js

  ```
  Then add `mentions` as a literal in the `currentStory` build object after `has_originated_relatives` (`index.html:1533`):
  ```js

  ```
  (`...story` is spread first at `index.html:1490` and never carries `mentions`, so the literal wins — same as the kinship-kernel literals at `index.html:1504-1514`.)

- **Runs once per scan, after research, regardless of save.** Like `resolveSymbolMeanings` it is NOT scan-limit-gated and non-billable to the user; it fires before the user reaches Result, independent of whether they later tap Save/Discard.

**Mobile twin:** same flow in `mobile/src/screens/CameraScreen.js` — build `mentionHits` from the per-source result vars after the parallel research fan-out (step 5 of the mobile pipeline), call `resolveMentions(mentionHits, graveData.subjects)`, attach the result as `mentions` on the story object that navigates to Result with `_unsaved: true`. It survives `handleSave` (plain key, not `_`-prefixed).

## 3. Raw fields to capture before discard, per source

The point where every raw var coexists is the `else` research branch, `index.html:1220-1376`. After line 1378 `searchResults`/`chronResults`/`archiveResults`/`wikiSummaryResults` go out of scope (block `const`s). Build `mentionHits` (a `buildMentionHits(...)` helper) there. Normalize all sources into a uniform `{ url, kind, snippet, year, source }`:

| Source | Var (line) | Raw shape | url | kind | snippet | year | source |
|---|---|---|---|---|---|---|---|
| Tavily | `searchResults` (`index.html:1220`) + `fgDetail` (`index.html:1234`) | `{title, url, content, source_type}` (`js/api-tavily.js:317-327`, `:388-392`) | `.url` | `"a present-day web page"` (or `"a FindAGrave memorial"` when `source_type==='memorial'`) | `.content` (≤6000) | **none** → `null` | `'web'` |
| Chronicling America | `chronResults` (`index.html:1223`) | `{title:'Paper (YYYY-MM-DD)', url, content, source_type:'public_domain'}` (`js/api-chroniclingamerica.js:44-49`) | `.url` | `"a Chronicling America newspaper page"` | `.content` (≤800) | parse `\d{4}` from `.title` | `'chronicling'` |
| Internet Archive | `archiveResults` (`index.html:1224`) | `{title:'Book (YYYY)', url:'archive.org/details/<id>', content, source_type:'archive'}` (`js/api-internetarchive.js:75-79`) | `.url` | `"an Internet Archive book or county history"` | `.content` (≤800) | parse `\d{4}` from `.title` | `'archive'` |
| Wikipedia | `wikiSummaryResults` (`index.html:1225`) → may also be the bridged/recovered object (`index.html:1291-1303`, `:1349-1352`) | `{title, extract, …}` (`js/api-wikipedia.js:107-110`) — **no `url`, no `content`** | **construct** `https://en.wikipedia.org/wiki/${encodeURIComponent(title.replace(/ /g,'_'))}` from `.title` | `"a Wikipedia article"` | `.extract` (≤2000) | **none** → `null` | `'wikipedia'` |

Normalization gotchas (all confirmed in the probe):
- **No uniform date field.** CA/IA embed the year only inside the `title` string — regex `\d{4}` out of `title`. Tavily/FindAGrave have NO date — `year: null`. Wikipedia has no date in the consumed object — `year: null`.
- **Wikipedia uses `extract` not `content`, and has NO url** — map `extract → snippet` and synthesize the article URL from `title`.
- **`wikidataResult`** (structured, no url/snippet — `js/api-wikidata.js:178-184`) and **WikiTree** (corroboration object, not a web hit) are **NOT** mention sources — exclude them.
- Skip the synthetic `WikiTree family record` source pushed at `index.html:1269-1276` (it carries `url:'https://www.wikitree.com/'` and a relationship-only description — not a real per-person mention; filter by `source_type === 'wikitree'`).
- Drop any hit whose `url` doesn't `startsWith('http')` BEFORE sending to Gemini (saves tokens; the validator drops them anyway).

## 7. Privacy & quality guardrails

- **Name-safe sentences (privacy):** Gemini authors every label under the S62 rule pair — name only confirmed-deceased subjects from the allowlist; any other person → relationship-only or omitted; *"when in doubt, treat as LIVING and remove the name."* (`js/api-gemini.js:374-428`). Because we control the label text, no raw snippet (which could name a living relative) ever reaches the UI.
  - **Backstop (optional but recommended):** if a mention sentence can derive a relative name from genealogy data, mirror the deterministic `stripOriginatedNamesForPublic` (`js/api-gemini.js:298`) as a fail-CLOSED post-filter rather than relying solely on the fail-OPEN Gemini pass. For the four chosen sources (web/CA/IA/Wikipedia) this is low-risk, but the strip is cheap insurance.
- **Wrong-person / common-surname relevance:** the biggest quality risk is a bare-surname hit ("SMITH") pointing at the wrong person. Mitigations:
  1. The prompt instructs the model to return **null** for any item that doesn't clearly refer to a deceased subject (the null-guard drops it).
  2. Pre-filter `rawHits` before the call by reusing the pipeline's existing name signal: prefer hits already in `mergedSearchResults` that survived FindAGrave `/extract` confirmation (`extractFindAGraveDetail`, `index.html:1234`) and de-prioritize bare-surname Tavily hits. (A future enhancement could reuse `_buildCorroborationSummary`'s name-match logic from `biography.js` as a relevance score — flag as an open option, not required for v1.)
  3. **Cap** the number of mentions: send at most ~8 hits to Gemini, and render at most **5** mentions (drop overflow). Keeps the sheet skimmable and the token cost bounded.
- **Link-rot:** store the captured `sentence` (and `year`) on the story. If the `url` later dies or fails the `startsWith('http')` guard, the UI renders the sentence as non-clickable text (`.mention-line-dead` web / `disabled` row mobile) — it still reads sensibly ("Mentioned in a 1919 obituary in the Aiken Courier.") without a working link.
- **Cost:** one extra `gemini-3.1-flash-lite` call per scan, NOT scan-limit-gated and non-billable to the user, fully overlapped with research (started inside the research branch, awaited at the existing merge point). Returns `[]` instantly when `rawHits` is empty (clean/dry scans cost nothing).
